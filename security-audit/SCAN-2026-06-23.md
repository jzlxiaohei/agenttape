# TraceLab 开源前安全扫描报告

- 扫描日期：2026-06-23
- 扫描对象：`a12b699` 加当前未提交工作树
- 运行模型：用户本机运行的 Go HTTP 服务与 React Viewer
- 结论：**可以继续推进开源；建议公开前优先处理 SA-01，并处理或明确接受 SA-03、SA-06。**

## 摘要

| 等级 | 数量 | 结论 |
| --- | ---: | --- |
| Critical | 0 | 未确认可直接利用的 Critical 问题 |
| High | 1 | localhost HTTP API 信任边界 |
| Medium | 2 | 凭证落盘、真实会话夹具隐私 |
| Low | 3 | 开发依赖、资源限制、浏览器防御头 |
| Informational / Accepted | 2 | 受信任用户的 Shell 能力、开源治理文件 |

本报告按以下主要场景评级：用户信任自己在 TraceLab UI 中输入的内容；用户本机上的普通进程不作为攻击者（它本来就拥有当前用户的大量权限）；服务默认只监听 loopback；用户仍可能访问恶意互联网网页，导入或展示不可信 Trace。若未来支持局域网或公网访问，需要重新提高多项评级。

项目默认监听 `127.0.0.1:8787`，这是正确的基础默认值；但浏览器中的 localhost 仍可能受到 DNS rebinding 等攻击。当前服务可以读取完整 Trace，并使用内存凭证重放计费请求，因此统一的 Host/Origin/实例授权仍是最重要的问题。

## Findings

### SA-01 [High]：危险的本地 HTTP API 没有统一访问控制

**证据**

- `cmd/tracelab/serve.go:17` 默认监听 loopback，但允许任意 `-listen`；`cmd/tracelab/serve.go:21` 的进程启动能力默认开启。
- `internal/server/server.go:44-48` 和 `internal/server/server.go:53` 直接暴露原始 mux，没有统一的 Host、Origin、CSRF 或实例令牌中间件。
- `internal/server/launch_api.go:19-32` 只检查 Origin 的主机名：任意端口的 `localhost` 都被信任，无 Origin 请求直接放行。
- Origin 检查只用于 launch/manual/install/restore。`internal/server/api.go:53-94` 的重放、计费 case run、删除和配置变更接口没有同类检查。
- `/_register` 接受任意 upstream，`/_sessions` 返回活动 session 与代理 token（`internal/server/server.go:81-100`）。

**影响**

- 在默认 loopback 模式下，DNS rebinding 或其他浏览器到 localhost 的访问路径可能读取 Trace、修改数据、注册代理并触发重放/计费调用。
- 如果用户把服务绑定到 `0.0.0.0` 或局域网地址，远程非浏览器调用不带 Origin，会直接通过 launch/config 修改接口的检查，后果可达到本机命令执行或配置篡改。
- 只给部分端点加 Origin 判断会留下新的状态变更接口漏保护的空间。

**建议**

1. 为整个服务增加统一中间件：校验 Host，并要求每个实例随机生成的 bearer/CSRF token。
2. 浏览器 API 要求 Origin 精确匹配实际 Viewer origin（含 scheme、host、port），不要只比较 hostname。
3. CLI/Hook/代理端点使用独立的随机凭证，不依赖“没有 Origin 就可信”。
4. 默认 `-allow-launch=false`；非 loopback 监听必须使用显式危险开关、强认证和醒目警告。
5. 所有状态变更与付费请求端点统一经过同一策略，并增加 DNS rebinding/CSRF 回归测试。

### SA-02 [Accepted capability]：受信任用户的额外参数由 Bash 解释

**证据**

- `internal/server/launch_api.go:161` 接收字符串类型的 `args`。
- `internal/server/launch_api.go:230-236` 把该字符串原样拼接到命令，并生成 Bash 脚本执行；只有可执行文件、工作目录和部分内部参数经过 `shellQuote`。
- 因此 `args` 中的 `;`、换行、重定向、命令替换等会被 Shell 解释，而不只是传给 Claude/Codex 的参数。
- 临时脚本在 `internal/server/launch_api.go:237-255` 创建后没有删除。

**场景判定**

在主要使用场景中，填写参数的本机用户是可信的；用户本来也可以打开终端执行同样的命令。因此这不是独立安全漏洞，而是一项有意或可接受的高级能力。它只有在 SA-01 被绕过、服务被绑定到非 loopback、或未来允许不可信调用方访问 launch API 时，才会变成命令执行漏洞。

**建议**

- 保留该能力也可以，但 UI/文档应说明参数按 Shell 语法解释，而不是普通 argv 数组。
- 该接口必须受 SA-01 的统一访问控制保护，非 loopback 模式不得无认证开放。
- 临时脚本执行后清理属于发布卫生改进。
- 如果未来希望接收不可信参数，再改为结构化 argv；当前不是开源阻塞项。

### SA-03 [Medium]：凭证头脱敏规则不完整，敏感数据文件权限过宽

**证据**

- `internal/source/httpcap/headers.go:22-29` 使用精确 denylist。它覆盖 `authorization`、`x-api-key` 和 Anthropic，但遗漏常见的 `api-key`、`x-goog-api-key`、`*-token`、`*-secret` 等。
- 项目允许自定义 upstream，因此未知提供商的认证头可能被写入 SQLite/JSONL。
- `internal/store/store.go:35-40` 创建数据目录和 raw 目录为 `0755`；`internal/store/rawfiles.go:28,43` 创建 session 目录为 `0755`、原始 Trace 为 `0644`。
- `internal/sink/jsonl.go:19` 创建 JSONL 为 `0644`。这些文件可能包含 Prompt、响应、工具输入、源码片段和未识别的凭证。

**影响**

在多用户机器或权限较宽的父目录中，同机其他账号可能读取 Trace；自定义提供商凭证可能被永久写入磁盘。

**建议**

- 复用 `case_curl.go` 已有的宽泛敏感头判断，至少匹配 auth/key/token/cookie/secret，并允许用户扩展规则。
- 更稳妥的方案是只持久化明确允许的安全头，而不是不断扩充 denylist。
- 数据目录使用 `0700`，数据库、JSONL 和 raw 文件使用 `0600`；打开已有文件后也校正权限。
- 文档明确正文不做自动脱敏；考虑提供可配置的 JSON 路径/正则脱敏层。

### SA-04 [Low]：Vitest 开发依赖包含官方 Critical 漏洞

**证据**

- `frontend/package.json:47` 声明 `vitest ^2.1.8`，锁文件实际安装 `vitest@2.1.9`。
- `npm audit` 报告 5 个开发依赖问题：1 Critical、1 High、3 Moderate。
- 其中 `GHSA-5xrq-8626-4rwp` 影响 Vitest `<3.2.6`：Vitest UI server 监听时可导致任意文件读取和执行。
- 依赖链还带入 `vite@5.4.21`、`esbuild@0.21.5` 的开发服务器问题。
- `npm audit --omit=dev` 对 175 个生产依赖报告 0 个漏洞。

**影响**

该问题不进入生产 Viewer 构建，当前 `vitest run` 脚本也不启动受影响的 UI server。在本项目实际调用路径下没有确认可利用性，因此项目评级为 Low；npm/上游公告的组件评级仍是 Critical。它适合尽快升级，但不是本地运行用户的发布阻塞项。

**建议**

- 升级到已修复的 Vitest 版本并更新锁文件；npm 当前建议 `vitest@4.1.9`，这是主版本升级，需要跑完整测试。
- 不把 Vitest UI/dev server 暴露到非 loopback 地址。

### SA-05 [Low]：请求、响应和 Hook 全量读入内存，HTTP Server 缺少资源限制

**证据**

- `internal/source/httpcap/proxy.go:52` 无限制读取请求体；`streamCopy` 同时把完整响应缓存在内存。
- `internal/source/hook/hook.go:41` 无限制读取 Hook body。
- `internal/server/replay.go:118,153` 无限制读取重放响应；返回值虽截断到 64 KiB，但完整响应已先进入内存。
- `cmd/tracelab/serve.go:57` 使用裸 `http.ListenAndServe`，没有 ReadHeaderTimeout、MaxHeaderBytes 等限制。

**影响**

异常客户端、Hook 或模型响应可显著占用内存/磁盘并拖垮本地进程。在受信任的本地输入场景中它主要是健壮性问题；服务一旦被暴露到局域网，风险会升高。

**建议**

- 对控制 API 和 Hook 使用 `http.MaxBytesReader`，按场景设置独立上限。
- 捕获流写入有大小上限的文件/缓冲；超限后继续转发但停止捕获，并记录 truncated。
- 使用显式 `http.Server` 配置 header/body/idle 超时与 MaxHeaderBytes。

### SA-06 [Medium]：测试夹具来自真实捕获，会携带环境信息和不透明会话数据

**证据**

- 请求与 seed 夹具合计约 1.44 MB，包含完整系统提示、工具输出和 OpenAI `encrypted_content`。
- 扫描发现本机路径用户名 `/Users/bytedance`；邮箱只有 `example@example.com` 和 `noreply@anthropic.com`。
- 通用秘密正则在高熵内容中命中两个 `sk-…` 片段。结构定位表明它们位于完整工具输出/加密内容内部，是随机高熵子串，**未确认是 API Key**。
- 未发现 Authorization、API Key、Cookie 等 JSON 凭证字段；已知密钥格式扫描没有确认真实凭证。
- 这些夹具从初始提交 `e1c7944` 起已存在于 Git 历史。

**影响**

即使没有可用密钥，真实捕获仍可能泄露用户名、工作方式、内部 Prompt、文件内容或未来未被规则识别的数据；高熵内容也会持续触发仓库秘密扫描告警。

**建议**

- 用最小化、合成的固定夹具替换真实会话，只保留测试所需字段。
- 删除 encrypted content、绝对路径、无关工具输出和内部提示。
- 如果仓库尚未公开，替换后评估是否清理 Git 历史；任何历史改写需单独备份和确认。

### SA-07 [Low]：Viewer 缺少浏览器安全响应头

`writeJSON` 只设置 Content-Type，静态 Viewer 也没有统一 CSP、`X-Content-Type-Options`、frame 限制和 Referrer Policy。当前 Markdown 使用 `react-markdown` 且没有启用 raw HTML，扫描未发现 `dangerouslySetInnerHTML`、`innerHTML`、`eval` 等危险 sink，这是积极控制；仍建议增加安全响应头，降低未来渲染回归的影响，并避免从本地 URL 泄露 Referrer。

### SA-08 [Informational]：缺少开源安全治理文件和自动检查

仓库未发现 `SECURITY.md`、GitHub Actions 安全扫描配置和 LICENSE。LICENSE 不属于代码漏洞，但会阻碍规范开源。建议加入私下报告渠道、支持版本、安全基线，以及 CI 中的秘密扫描、`govulncheck`、`npm audit`/Dependabot 和静态检查。

## 已通过或积极控制

- 服务默认监听 `127.0.0.1:8787`。
- Session ID/token 使用 `crypto/rand`；没有使用时间戳或可预测随机数。
- 已知 OpenAI/Anthropic 认证头在持久化前脱敏，API Key launch 模式把真实 Key 留在进程内存。
- Markdown 不解析 raw HTML；前端静态扫描未发现直接 HTML/脚本执行 sink。
- SQL 搜索条件通过参数绑定，未发现用户值直接拼接到 SQL。
- `go test ./...` 通过；`go vet ./...` 通过（使用一致的 Go 1.26.4 工具链）。
- `govulncheck`：可达代码 0 漏洞；依赖模块有 1 个不可达且仅 Windows 的 `GO-2026-5024`（`x/sys v0.42.0`，修复于 `v0.44.0`）。
- 前端生产构建通过；生产 npm 依赖漏洞为 0。

## 验证异常与限制

- 默认 `go` 命令为 1.26.2，但 GOROOT 指向 1.26.4，导致版本不匹配；切换到 `/Users/bytedance/.gvm/gos/go1.26.4/bin/go` 后测试和 vet 通过。这是本机工具链问题，不是项目代码漏洞。
- 前端 22 个测试中 21 个通过，`ContentBlocks` 的 Markdown lazy-render 测试失败；生产构建成功。该失败未证明安全漏洞，但发布前应恢复全绿。
- 本轮没有主动执行命令注入、付费重放或修改用户配置，只做静态验证和无副作用测试。
- 环境没有安装 gitleaks/trufflehog/gosec；秘密扫描使用已知凭证格式、敏感字段检查和 Git patch 历史扫描。发布前建议再用专用秘密扫描器复核最终历史与发布包。

## 建议修复顺序

1. 统一 localhost API 授权与 Host/Origin 策略；非 loopback 模式必须另行认证。
2. 完善敏感头规则并收紧数据文件权限。
3. 合成并最小化测试夹具，随后复扫 Git 历史与最终发布物。
4. 明确记录 `args` 使用 Shell 语法；保留该能力不阻塞开源。
5. 升级 Vitest/锁文件，恢复所有测试通过。
6. 增加 body/response 限制、HTTP Server 超时和安全响应头。
7. 增加 `SECURITY.md`、LICENSE 和 CI 安全检查。
